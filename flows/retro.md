# `retro` 阶段

## 目的

让一次运行沉淀为流程学习，而不只是留下代码产物。

## 输入

- 各阶段输出
- `review` 失败记录
- 验证证据
- 返工点

## 必需输出

- 保留什么
- 修改什么
- 移除什么
- 针对技能、文档或检查的候选改进

## Gate

- 至少存在一条具体的流程学习；若无需流程改动，也必须明确说明原因
- 学习点指向未来工作流质量，而不是营销式总结

## 来源视角

- `gstack`
  - 使用 `retro` 的流程学习视角，总结哪些约束、检查和切分方式有效。
- `superpowers`
  - 允许整理运行痕迹、返工点和学习项。
  - 不允许借复盘直接改写规则，也不允许把 backlog 伪装成 retro。
- `harness engineering`
  - 要求 `retro` 成为强制收尾，并把失败模式转化为后续 guard / docs / eval 候选改进。

## 失败条件

- 空洞口号
- 把产品 backlog 伪装成 retro
- 未经审查的流程重写
- 只说“下次更小心”却没有结构性动作

## 失败去向

无。`retro` 是强制终止阶段。
