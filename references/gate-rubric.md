# Gate 准绳

将本文件作为所有阶段共享的 go / no-go 评分准绳。

## 通用 Go 问题

只有以下问题全部回答为“是”，阶段才能通过：

1. 当前阶段所需输入是否显式且已存在？
2. 当前阶段输出是否足够显式，使下一阶段无需猜测就能消费？
3. 当前范围是否仍然被约束在单个任务内，并在需要时保持为单个活动 ticket？
4. 相关 guard 是否都已检查并满足？
5. 如果阶段未通过，失败去向是否显式？

只要有任何一个回答为“否”，该阶段就不能通过。

## 各阶段 Go 问题

### `reframe`

- 是否只有一个主目标？
- 成功标准之后是否可验证？
- 非目标和假设是否显式？

### `plan`

- 当前是否只有一个方案？
- `build` 前是否已存在验证契约？
- 影响区域是否受控？

### `ticket`

- 是否只有一个活动 ticket？
- 这个 ticket 是否足够小，能在一次 `build -> review` 循环中关闭？
- 验收点和停止条件是否显式？

### `build`

- 每条变更是否都能映射到活动 ticket？
- 是否不存在 顺手夹带的额外内容？
- 在进入 `review` 前，明显错误状态是否已经清除？

### `review`

- 是否不存在未解决的高严重度问题？
- Scope drift 是否不存在，或已被显式判失败？
- 隐藏回退是否不存在，或已被显式判失败？

### `verify`

- 每个验收点是否都有直接证据，或已被显式标记为 `blocked`？
- 必需检查是否真的执行过，或其未执行原因被显式正当化？
- 结论是否与证据完全一致？

### `retro`

- 是否至少存在一条具体流程学习，或有合理说明表明无需改动？
- `retro` 是否避免了产品 backlog 替代和纯口号输出？

## 声明类型与证据类型分类

### 静态结构声明
- 可接受证据：文件内容、diff、配置文本、明确的静态检查输出

### 行为声明
- 优先要求：执行得到的检查结果
- 环境缺失时：只能 `blocked` 或 `unverified`

### 外部环境声明
- 可接受：用户提供的外部证据，必须标注来源

## No-Go 结果

如果某阶段未通过 rubric：

- 返回所需的更早阶段，或
- 输出 `blocked`

绝不允许只靠信心继续推进。
