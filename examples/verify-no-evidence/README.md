# Verify 缺乏证据示例（负例）

这个示例展示了 `verify` 阶段因缺乏直接行为证据而失败，标记 unverified 项并回退到 `build` 补充验证产物的合规处理路径。它是一个**负例**，演示流程如何拒绝无证据完成。

## 来源系统体现

- `gstack`：固定阶段机完整执行，`verify` 用验证视角逐条检查证据映射。
- `superpowers`：在活动 ticket 内执行检查和收集证据，不用更弱检查替代必需检查。
- `harness engineering`：`verify` gate 要求证据优先，证据不足时回退 `build` 补充测试，`retro` 强制收尾。

## `reframe`

任务陈述:
将一个配置文件的解析逻辑从 JSON 迁移到 YAML，保持运行时行为不变。

成功标准:
- YAML 配置被正确解析
- 运行时行为与 JSON 版本一致

非目标:
- 不重新设计配置结构
- 不增加新配置项

假设:
- 现有 JSON 配置的语义可以 1:1 映射到 YAML

未知项:
- 现有测试是否覆盖了所有配置路径

阻塞项:
- 无

完成定义:
ticket 被实现、review、verify 并以 `retro` 收尾。

## `plan`

已选方案:
替换解析器，保持配置语义不变；用解析回归测试验证 YAML 能被正确解析；再用代表性配置样本的行为对比检查验证运行时行为等价。

被拒绝的替代方案:
- 同时支持 JSON 和 YAML（超出范围）
- 重写配置加载层（过度工程）

影响区域:
- 配置解析模块
- 配置文件
- 解析回归测试文件
- 代表性配置样本清单

主要风险:
- YAML 解析器对某些边界值的处理可能与 JSON 不同

验证契约:
- acceptance: YAML 配置被正确解析
  evidence: 解析回归测试结果
- acceptance: 运行时行为与 JSON 版本一致
  evidence: 代表性配置样本上的行为对比检查输出

回滚思路:
如果行为差异无法消除，回滚到 JSON 解析。

## `ticket`

Ticket 编号: T1
目标: 将配置解析从 JSON 迁移到 YAML
边界: 仅限解析模块、配置文件、解析回归测试和代表性配置样本清单
输入: 已批准的 plan
预期产物: YAML 解析器替换、解析回归测试和行为对比所需样本
验收点:
- YAML 配置被正确解析
- 行为与 JSON 版本一致
停止条件:
如果发现语义无法 1:1 映射，返回 `plan`。

当前 Ticket: T1

变更文件:
- 路径: 配置解析模块
  原因: 替换 JSON 解析为 YAML 解析
- 路径: 配置文件
  原因: 从 .json 转为 .yaml 格式
- 路径: 解析回归测试文件
  原因: 验证 YAML 配置能够被正确解析

新发现问题:
- 尚未准备代表性配置样本，也未产出行为对比检查结果

## `review`

状态: 通过

问题:
- 严重度: 无
  问题: 无
  修复: 无

范围漂移:
- 无

隐藏回退:
- 无

## `verify`（第 1 轮 — 失败）

验收点 -> 证据:
- YAML 配置被正确解析 -> 解析回归测试通过
- 运行时行为与 JSON 版本一致 -> **缺失：未产出代表性配置样本上的行为对比检查结果**

已执行检查:
- 命令: 解析回归测试
  结果: 通过
- 命令: 现有测试套件
  结果: 通过（但不包含 YAML 与 JSON 的行为对比检查）

未验证项:
- 行为等价性（无直接对比证据）

阻塞项:
- 无

剩余风险:
- 现有测试通过不等于 YAML 行为等价

结论:
**失败。证据不足，无法宣称 verified。回退到 `build` 补充行为对比样本和检查结果。**

## `build`（第 2 轮 — 补充验证产物）

当前 Ticket: T1

变更文件:
- 路径: 代表性配置样本清单
  原因: 补充 JSON 与 YAML 共用的代表性配置样本
- 路径: 行为对比检查文件
  原因: 产出 JSON 到 YAML 的行为对比结果，覆盖正常值和边界值

新发现问题:
- 无

## `verify`（第 2 轮 — 通过）

验收点 -> 证据:
- YAML 配置被正确解析 -> 解析回归测试通过
- 运行时行为与 JSON 版本一致 -> 行为对比检查覆盖正常值和边界值，输出一致

已执行检查:
- 命令: 解析回归测试
  结果: 通过
- 命令: 行为对比检查
  结果: 通过
- 命令: 现有测试套件
  结果: 通过

未验证项:
- 无

阻塞项:
- 无

剩余风险:
- 极端边界值（如深层嵌套、特殊字符）未完整覆盖

结论:
`verified`

## `retro`

保留:
- `verify` 拒绝在无证据时宣称完成，有效防止了假阳性

修改:
- `build` 阶段应尽早准备 verify 所需的直接证据，不要把关键验证产物拖到最后才补

移除:
- "现有测试通过就等于行为等价"的思维惯性

流程后续:
- 考虑在 `build` gate 中增加"verify 所需证据是否已具备最小采集条件"检查项

已验证:

- 行为与 JSON 版本一致

- 极端边界值覆盖未完整验证
