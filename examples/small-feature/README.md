# 小型功能示例

这个示例展示了一个小型功能新增的合规运行方式。

## 来源系统体现

- `gstack`：用固定阶段机推进，并在 `plan` 中用价值、范围和技术风险视角收敛方案。
- `superpowers`：只在已批准方案和活动 ticket 内做文件编辑、测试执行和上下文整理。
- `harness engineering`：要求功能新增仍然遵守证据优先、显式回归保护和强制 `retro`。

## `reframe`

任务陈述:
增加一个 `--json` CLI 开关，使某个命令可以为下游工具输出结构化结果。

成功标准:
- 命令接受 `--json`
- 开关存在时输出有效结构化 JSON
- 现有默认文本输出保持不变

非目标:
- 不做 CLI 重设计
- 不新增子命令

假设:
- 当前命令已经有可直接序列化的稳定输出结构

未知项:
- 现有 CLI 测试是否覆盖开关解析

阻塞项:
- 无

完成定义:
开关被实现、review、verify 并以 `retro` 收尾。

## `plan`

已选方案:
增加一个窄范围的开关解析分支和一条聚焦序列化路径，再用定向 CLI 测试验证。

被拒绝的替代方案:
- 重写整个格式化器
- 增加通用输出插件系统

影响区域:
- CLI 参数解析器
- 命令输出格式化器
- 聚焦 CLI 测试

主要风险:
- 破坏默认文本输出

验证契约:
- acceptance: `--json` 输出有效 JSON
  evidence: 定向 CLI 测试输出
- acceptance: 默认输出仍然是文本
  evidence: 定向回归测试输出

回滚思路:
如果开关分支改变了默认输出行为，就回滚它。

## `ticket`

Ticket 编号: T1
目标: 在不改变默认输出模式的前提下增加 `--json`
边界: parser 分支、格式化器分支、定向测试
输入: 已批准的 plan
预期产物: 最小功能差异与聚焦测试
验收点:
- `--json` 输出结构化 JSON
- 默认输出保持不变
停止条件:
如果必须重设计输出结构，返回 `plan`。

## `build`

当前 Ticket: T1

变更文件:
- 路径: CLI parser
  原因: 识别 `--json`
- 路径: 格式化器
  原因: 为 JSON 模式序列化现有输出
- 路径: CLI tests
  原因: 覆盖 json 模式与默认文本模式

新发现问题:
- 无

## `review`

状态: 通过

问题:
- 严重度: 低
  问题: 确保 JSON 模式不会暗中变成默认模式
  修复: 显式验证两种模式

范围漂移:
- 无

隐藏回退:
- 无

## `verify`

验收点 -> 证据:
- `--json` 输出结构化 JSON -> 定向 CLI 测试通过并校验 JSON 输出
- 默认输出保持不变 -> 文本模式定向回归测试通过

已执行检查:
- 命令: 聚焦 CLI 测试命令
  结果: 通过

未验证项:
- 本示例未运行完整端到端集成链路

阻塞项:
- 无

剩余风险:
- 与下游工具的完整集成仍未验证

结论:
`verified`

## `retro`

保留:
- 将新行为和回归保护拆成独立验收点

修改:
- 在功能示例中继续强调对默认模式回归的显式验证

移除:
- 任何把功能验证压缩成单一路径成功用例测试的冲动

流程后续:
- 继续让功能示例至少包含一个回归型验收点

## 交接

改动内容:

- 增加 json 模式和默认模式的聚焦测试

已验证:
- json 模式输出
- 默认文本模式未变化

阻塞或未验证:
