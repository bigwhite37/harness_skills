# 安全重构示例

这个示例展示了“无预期行为变化”的安全重构应如何合规运行。

## 来源系统体现

- `gstack`：用固定阶段机推进，并把 `review` 和 `verify` 分开处理“是否漂移”和“是否有证据证明未变”。
- `superpowers`：只在活动 ticket 内做受控重构和回归检查，不借重构名义扩范围。
- `harness engineering`：要求安全重构必须留下“行为未变”的直接证据，否则不能宣称完成。

## `reframe`

任务陈述:
将重复的 normalization 逻辑抽取为共享 helper，且不改变运行时行为。

成功标准:
- 行为保持不变
- 重复逻辑被一个共享 helper 消除
- `verify` 产出行为未变化的直接证据

非目标:
- 不改变算法
- 不改变数据模型

假设:
- 当前重复路径在行为上本就一致

未知项:
- 现有测试是否完整覆盖两个调用点

阻塞项:
- 无

完成定义:
重复逻辑被移除，`review` 确认没有 范围漂移，`verify` 证明行为未变，并完成 `retro`。

## `plan`

已选方案:
抽取一份逻辑完全相同的共享 helper，并使用覆盖两个现有调用点的聚焦回归测试进行验证。

被拒绝的替代方案:
- 扩展为所有相关模块的大清理
- 会改变行为的 normalization 重设计

影响区域:
- 两个调用点
- 一个新的共享 helper
- 回归测试

主要风险:
- 抽取过程中意外引入行为漂移

验证契约:
- acceptance: 两个调用点保持当前输出
  evidence: 两条路径的聚焦回归测试

回滚思路:
如果回归证据失败，就把 helper 重新内联。

## `ticket`

Ticket 编号: T1
目标: 安全抽取重复 normalization 逻辑
边界: 两个调用点、一个共享 helper、聚焦回归测试
输入: 已批准的 plan
预期产物: 不改变行为的 helper 抽取
验收点:
- 重复逻辑被一个 helper 替换
- 两个调用点保持当前行为
停止条件:
如果需要改变行为才能完成重构，返回 `plan`。
