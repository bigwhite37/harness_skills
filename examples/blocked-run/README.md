# 阻塞运行示例

这个示例展示了当验证无法被诚实完成时，一个合规的 `blocked` 结果应该是什么样。

## 来源系统体现

- `gstack`：固定阶段机仍然完整执行到 `verify` 与 `retro`，不会因为阻塞而跳过流程解释。
- `superpowers`：在发现环境不可用后停止推进，不用更弱替代检查冒充完成。
- `harness engineering`：把失败转成显式 `blocked`，并要求恢复动作和剩余风险都被写清楚。

## `reframe`

任务陈述:
确认一项配置迁移在运行时保持原有行为。

成功标准:
- 在真实执行环境中验证迁移行为

非目标:
- 不做配置重设计

假设:
- 只有真实执行环境才能提供可信验证

未知项:
- 本地是否能访问所需环境

阻塞项:
- 可能缺少环境访问能力

完成定义:
要么存在直接运行时证据，要么本次运行以 `blocked` 结束。

## `plan`

已选方案:
先完成迁移变更，再在所需运行时环境中验证。

被拒绝的替代方案:
- 靠目测假设行为等价
- 用更弱的本地 冒烟检查 替代

影响区域:
- 迁移文件
- 运行时验证路径

主要风险:
- 行为差异可能只会在真实环境中暴露

验证契约:
- acceptance: 迁移后的配置在运行时行为不变
  evidence: 运行时验证结果

回滚思路:
如果运行时验证失败，则回滚迁移。

## `ticket`

Ticket 编号: T1
目标: 为运行时验证准备迁移
边界: 仅限迁移变更
输入: 已批准的 plan
预期产物: 候选迁移变更
验收点:
- 可以尝试运行时验证
停止条件:
如果运行时环境不可用，停止并输出 `blocked`。

## `build`

当前 Ticket: T1

变更文件:
- 路径: 迁移文件
  原因: 准备运行时验证所需的候选迁移

新发现问题:
- 当前上下文中无法访问所需运行时环境

## `review`

状态: 通过

问题:
- 严重度: 无
  问题: 无
  修复: 无

范围漂移:
- 无

隐藏回退:
- 无

## `verify`

验收点 -> 证据:
- 迁移后的配置在运行时行为不变 -> 缺失

已执行检查:
- 命令: 运行时验证命令
  结果: 未运行，所需环境不可用

未验证项:
- 运行时行为等价性

阻塞项:
- 无法获得直接运行时证据

剩余风险:
- 即使本地目测通过，真实环境中仍可能出现行为差异

结论:
`blocked`

## 阻塞

阶段: `verify`
原因: 所需运行时环境不可用，导致验证契约无法满足
影响: 不允许宣称已验证完成
恢复动作: 在所需环境中重新执行 `verify`，或返回 `plan` 并改成新的已批准验证契约

## `retro`

保留:
- 拒绝用更弱替代检查冒充验证

修改:
- 当任务依赖运行时访问时，在 `reframe` 更早暴露环境要求

移除:
- 把本地目测当成验证的任何冲动

流程后续:
- 对依赖运行时环境的任务，在 `reframe` 的未知项里明确记录环境可用性
