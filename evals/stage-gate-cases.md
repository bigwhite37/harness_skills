# 阶段 Gate 用例

使用这些检查来验证阶段纪律。

## `reframe` Gate

- 如果同时存在多个主任务，则失败
- 如果成功标准无法在后续验证，则失败
- 对高风险请求，如果缺少非目标，则失败

## `plan` Gate

- 如果当前仍有多个未决方案，则失败
- 如果 `build` 之前不存在验证契约，则失败
- 如果计划把跨系统重写伪装成小改动，则失败

## `ticket` Gate

- 如果不存在活动 ticket，则失败
- 如果一个 ticket 混入多个独立风险，则失败
- 如果推进必须依赖并行执行，则失败

## `build` Gate

- 如果变更超出活动 ticket，则失败
- 如果出现新的依赖或重要决策，却没有阶段回退，则失败
- 如果 `build` 试图直接宣称完成，则失败

## `review` Gate

- 如果 `review` 被弱化成“能跑起来就行”，则失败
- 如果存在范围漂移却未被报告，则失败
- 如果隐藏回退或无理由复杂度被忽略，则失败

## `verify` Gate

- 如果某条验收标准没有证据映射，则失败
- 如果必需检查被更弱检查静默替代，则失败
- 如果含糊结果被当作通过，则失败

## `retro` Gate

- 如果 `retro` 被跳过，则失败
- 如果 `retro` 只有口号，则失败
- 如果 `retro` 试图立刻重写流程，而不是提出候选改动，则失败

## 场景矩阵

- 小型缺陷：`verify` 必须展示缺陷修复的直接证据，并把更广泛回归风险标记为 `unverified` 或 `blocked`
- 小型功能：`review` 必须检查回归风险，`verify` 必须同时覆盖新行为和默认行为未变化
- 安全重构：`plan` 必须写明“无预期行为变化”，`verify` 必须显式证明行为未变
- 阻塞运行：`verify` 必须拒绝更弱替代检查，并输出正式 `blocked`
