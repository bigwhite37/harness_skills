# 触发用例

## 使用方式

这些用例可用于手动对照验证或喂给模型做自动化检查：
- **手动对照**：将"用户 prompt"输入模型，检查模型是否按"预期行为"响应。
- **模型验证**：将用例格式化为 prompt-response 对，让评估模型判断技能是否在正确时机触发或拒绝触发。

---

## 覆盖说明

用例覆盖以下维度：
- **应触发**：非平凡缺陷修复、小型功能、安全重构、blocked 场景
- **不应触发**：纯解释、头脑风暴、部署操作、自治清理

## 必需场景覆盖

- 小型缺陷：技能应触发，并保持单个活动 ticket 与显式证据
- 小型功能：技能应触发，并保留对现有行为的回归覆盖
- 安全重构：技能应触发，并要求"行为未变"的证据
- 阻塞运行：当契约无法满足时，技能应触发并显式以 `blocked` 结束

## 结构化测试用例

### TC-01: 小型缺陷修复

- **id**: TC-01
- **input**: "这个函数在空输入时会崩溃，帮我修一下，要有测试"
- **expected**: 应触发
- **pass_criteria**: 技能从 `reframe` 开始，保持单个活动 ticket，`verify` 阶段要求直接证据
- **category**: 应触发 — 缺陷修复
- **rationale**: 非平凡缺陷修复，需要范围控制和证据式验证

### TC-02: 小型功能添加

- **id**: TC-02
- **input**: "给用户列表页加一个按注册时间排序的功能，不要改现有的排序逻辑"
- **expected**: 应触发
- **pass_criteria**: 技能触发并在 `plan` 中定义验证契约，`review` 检查是否影响现有排序
- **category**: 应触发 — 小型功能
- **rationale**: 需要阶段机保证新功能不破坏现有行为

### TC-03: 安全重构

- **id**: TC-03
- **input**: "把用户认证从 session 迁移到 JWT，要保证现有行为不变"
- **expected**: 应触发
- **pass_criteria**: `plan` 写明"无预期行为变化"，`verify` 显式证明行为未变
- **category**: 应触发 — 安全重构
- **rationale**: 安全重构需要阶段机保证行为等价性验证

### TC-04: 阻塞场景

- **id**: TC-04
- **input**: "修复生产环境的数据库连接池泄露，但我没有生产环境的访问权限"
- **expected**: 应触发，最终以 `blocked` 结束
- **pass_criteria**: 技能触发后在 `verify` 或更早阶段因环境不可用输出 `blocked`，不做静默降级
- **category**: 应触发 — blocked
- **rationale**: 缺少必需环境时必须显式 blocked，不能假装验证通过

### TC-05: 纯代码解释

- **id**: TC-05
- **input**: "帮我解释一下 src/parser.ts 的逻辑"
- **expected**: 不应触发
- **pass_criteria**: 技能不被触发，直接提供代码解释
- **category**: 不应触发 — 纯解释
- **rationale**: 纯代码解释无改动请求，不属于工作流管辖范围

### TC-06: 纯头脑风暴

- **id**: TC-06
- **input**: "我们讨论一下后端架构怎么演进比较好"
- **expected**: 不应触发
- **pass_criteria**: 技能不被触发，进行开放式讨论
- **category**: 不应触发 — 头脑风暴
- **rationale**: 没有边界任务的纯头脑风暴不适用固定阶段机

### TC-07: 部署操作

- **id**: TC-07
- **input**: "帮我把这个 PR 合并一下，然后部署到 staging"
- **expected**: 不应触发
- **pass_criteria**: 技能不被触发，告知用户部署操作不在技能管辖范围
- **category**: 不应触发 — 部署操作
- **rationale**: 部署/PR 管理操作不属于技能管辖范围

### TC-08: 自治清理

- **id**: TC-08
- **input**: "把整个项目里的 console.log 都清理掉，顺便统一一下代码风格"
- **expected**: 不应触发
- **pass_criteria**: 技能不被触发，而是先要求把请求收缩为单一任务边界，再决定是否进入该工作流
- **category**: 不应触发 — 自治清理
- **rationale**: 跨多个无关文件的开放式清理没有单一任务边界，不适用收敛式工作流

### TC-09: 有环境基线约束的小型功能

- **id**: TC-09
- **input**: "给这个 FastAPI 服务加一个软删除功能。项目要求 Python 3.10，测试数据库可能还是旧 schema。"
- **expected**: 应触发
- **pass_criteria**: 技能触发后在 `reframe` 或 `plan` 中显式锁定解释器、依赖安装方式和旧 schema 基线，不能直接默认环境正确
- **category**: 应触发 — 环境受限功能
- **rationale**: 非平凡功能 + 显式环境/状态前提，必须进入阶段机并先收敛前提条件

## 失败信号

- 技能在没有显式上游输出的情况下直接从 `build` 开始
- 技能被调用来处理发布或部署工作
- 技能不再像工作流治理器，而变得像通用自治智能体
